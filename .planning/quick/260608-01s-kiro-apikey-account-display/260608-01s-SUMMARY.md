# Quick Task 260608-01s Summary: Repair kiro api-key account display

## What was wrong
Commit `1d28d9f` enriched Kiro Quota Tracker cards but never made api-key kiro
accounts actually display. Two symptoms (kiro only):
1. Quota Tracker omitted `authType: "apikey"` kiro accounts entirely.
2. Providers tab kiro card counted 3 (oauth only) while the detail page showed 4.

## Changes

### `src/shared/constants/providers.js`
Added `"kiro"` to `USAGE_APIKEY_PROVIDERS`. This makes `isUsageEligible()`
(`src/app/api/providers/client/route.js:46`) and the usage route
(`src/app/api/usage/[connectionId]/route.js:136`) accept api-key kiro connections,
so they now surface on dashboard/quota with the same enriched card UI.

### `src/app/(dashboard)/dashboard/providers/page.js`
- `getProviderStats(providerId, authType)` now accepts a scalar OR an array of
  authTypes (`Array.isArray` normalization). Backward compatible — all existing
  scalar callers behave identically.
- The kiro free-provider card now renders with `["oauth", "apikey"]` for both its
  stats and its toggle, so the list count matches the detail page and the provider
  on/off toggle covers both connection kinds. Non-kiro free providers keep `"oauth"`.

## Verification
- ESLint on both changed files: my edited regions are clean. The 2 reported errors
  (`react-hooks/set-state-in-effect` at lines 872/1061) are pre-existing in the
  Add*CompatibleModal components — confirmed by linting the stashed (pre-change)
  file, which reports the same 2 errors. Not introduced by this task.

## Scope
Kiro provider only. No behavior change for other providers.
